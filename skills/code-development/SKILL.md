## 开发规范原则

- **参考现有代码风格**：保持与项目现有代码一致（缩进、命名、注释等）
- **优先复用而非新建**：使用已有的组件、工具函数
- **边界检查**：只修改允许修改的文件/目录
- **安全性**：避免注入、XSS 等常见漏洞
- **简洁性**：避免过度设计，直接满足需求

## 开发流程

### 阶段 1: 数据层

```
1. 创建/修改数据模型
2. 创建/修改请求/响应模型
3. 更新数据库 schema（如需要）
```

### 阶段 2: 业务层

```
1. 创建/修改业务逻辑服务
2. 添加外部服务调用
3. 实现核心功能逻辑
```

### 阶段 3: 接口层

```
1. 创建/修改 API Handler
2. 注册路由
3. 添加中间件（如需要）
```

### 阶段 4: 前端/界面（如适用）

```
1. 创建接口调用层
2. 创建/修改界面组件
3. 添加事件处理
```

## 开发中遇到阻塞的处理

当遇到问题时：

```
1. 先尝试自己解决
   - 查阅项目文档
   - 查看类似代码实现
   - 搜索相关技术资料

2. 短时间内无法解决
   - 记录问题到「待确认问题」
   - 继续开发其他独立任务

3. 统一询问
   - 将所有待确认问题汇总
   - 一次性向用户询问
```

### 待确认问题模板

```markdown
## 待确认问题

1. **[问题 1]**
   - 上下文：在实现 xxx 时
   - 需确认：xxx 接口返回格式是什么？

2. **[问题 2]**
   - 上下文：在定义 xxx 字段时
   - 需确认：最大长度是多少？
```

## 开发自检点

**每完成一个开发阶段，必须检查：**

```
□ 代码编译/构建通过
□ 没有调试代码残留（console.log、debugger 等）
□ 符合项目现有代码风格
□ 只修改了允许修改的文件
□ 没有引入安全漏洞
□ 报告当前进度
```

### 编译/构建命令示例

```bash
# 前端项目
npm run build
npm run lint

# 后端项目
go build ./...
mvn compile
```

## 注意事项

1. **不要过度设计**：按需求实现，避免添加"可能有用"的功能
2. **及时自检**：每个阶段完成后检查编译和风格
3. **记录阻塞**：遇到无法解决的问题及时记录，不要阻塞全部开发
4. **保持沟通**：定期报告进度，让用户了解开发状态
