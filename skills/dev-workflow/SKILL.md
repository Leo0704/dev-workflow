当收到开发需求时，必须严格按照以下工作流执行。

---

## ⚠️ 核心原则

> **需求理解不清楚，坚决不写代码**
> 宁可多花 30 分钟理解需求，也不要花 3 小时改错代码
> **用户没确认模糊点，坚决不进入下一步**

### 团队模式判断

> ⚠️ **重要：必须执行以下判断流程，不能跳过！**

**团队模式判断检查清单（满足任意2条即启用）：**

```
□ 涉及文件数量 > 5 个
□ 涉及模块/服务数量 > 2 个
□ 需要前后端同时开发
□ 需要调研 + 开发同时进行
□ 涉及外部系统集成 > 1 个
□ 预计代码行数 > 500 行
□ 任务描述超过 200 字
```

**判断输出格式（必须在需求理解报告中填写）：**

```markdown
## 团队模式判断

| 检查项 | 结果 | 说明 |
|--------|------|------|
| 涉及文件数 | ✅ >5 / ❌ ≤5 | 具体数量 |
| 涉及模块数 | ✅ >2 / ❌ ≤2 | 模块列表 |
| 前后端并行 | ✅ 是 / ❌ 否 | 说明 |
| 调研+开发 | ✅ 是 / ❌ 否 | 说明 |
| 外部集成 | ✅ >1 / ❌ ≤1 | 服务列表 |
| 代码规模 | ✅ >500行 / ❌ ≤500行 | 预估 |
| 任务复杂度 | ✅ 复杂 / ❌ 简单 | 说明 |

**判断结果:** ✅ 启用团队模式 / ❌ 单 agent 模式
**理由:** [具体说明]
```

**团队模式配置（如启用）：**

|------|----------|-----------|

## 任务目录结构

```
.claude/task/
├── 2024-01-15-用户冻结功能/    ← 任务目录（按日期+功能命名）
│   ├── prd.pdf                ← 需求文档
│   ├── 设计图.png
│   ├── test-plan.md           ← 测试计划
│   ├── test-cases.md          ← 测试用例
│   ├── test-report.md         ← 测试报告
│   └── .workflow-step         ← 工作流状态（自动生成）
├── 2024-01-20-订单导出/
│   └── ...
└── .current-task              ← 当前任务标识
```

**创建新任务：**
```bash
# 1. 创建任务目录
mkdir -p .claude/task/$(date +%Y-%m-%d)-功能名称

# 2. 设置为当前任务
echo "$(date +%Y-%m-%d)-功能名称" > .claude/task/.current-task

# 3. 放入需求文档
cp ~/Downloads/prd.pdf .claude/task/$(date +%Y-%m-%d)-功能名称/
```

## 工作流状态跟踪

**每进入一个新步骤，必须先更新状态文件：**

```bash
echo "N" > .claude/task/$(cat .claude/task/.current-task)/.workflow-step
```

工作流完成后删除：`rm .claude/task/$(cat .claude/task/.current-task)/.workflow-step`

## 步骤 0: 历史学习检查（推荐）

> 在开始任何任务前，先检查是否有相关的历史学习

**检查项目记忆和学习日志：**

```bash
# 读取项目记忆（跨会话持久化）
cat MEMORY.md

# 检查学习日志目录
cat .learnings/LEARNINGS.md
cat .learnings/ERRORS.md
```

**文件说明：**

| 文件 | 作用 |
|------|------|
| `MEMORY.md` | 项目级跨会话记忆，存储重要约定和规范 |
| `.learnings/LEARNINGS.md` | 学习记录，纠正和最佳实践 |
| `.learnings/ERRORS.md` | 错误记录和解决方案 |

**历史学习的价值：**
- 避免重复犯同样的错误
- 复用已验证的解决方案
- 遵循项目特定的最佳实践

### 1.5 特殊需求格式处理

**图片/原型图处理:**
- 优先使用图像分析 MCP（如 mcp__zai-mcp-server__ui_to_artifact）
- 描述图片中的：布局结构、字段名称、交互元素、操作按钮

1. 在报告中列出冲突点
2. 标注每份文档的说法

**需求变更时:**
1. 记录变更内容

### 3.2 风险等级定义
