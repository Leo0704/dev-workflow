## 测试文档编写

在任务目录下创建测试文档：

```
task/2024-01-15-功能名称/
├── test-plan.md          ← 测试计划
├── test-cases.md         ← 测试用例
└── test-report.md        ← 测试报告（执行后生成）
```

---

## 测试计划模板

**文件：`test-plan.md`**

```markdown
# 测试计划

## 1. 测试范围
| 模块 | 功能点 | 测试类型 |
|------|--------|----------|
| xxx | xxx | 单元测试/集成测试/E2E |

## 2. 测试策略

### 2.1 单元测试
- 覆盖率目标：>= 80%
- 重点测试：核心业务逻辑、边界条件、异常处理

### 2.2 集成测试
- 测试场景：API 接口、数据库交互、外部服务调用
- 数据准备：测试数据、Mock 数据

### 2.3 E2E 测试（如适用）
- 测试场景：完整业务流程
- 前置条件：测试环境、测试账号

## 3. 测试环境
| 环境 | 地址 | 用途 |
|------|------|------|
| 开发环境 | xxx | 开发自测 |
| 测试环境 | xxx | 集成测试 |

## 4. 风险评估
| 风险 | 影响 | 应对措施 |
|------|------|----------|
| xxx | 高/中/低 | xxx |
```

## 执行测试

根据项目类型选择测试命令：

```bash
# 前端项目
npm test / yarn test / pnpm test
npm run test:e2e  # E2E 测试

# 后端项目
go test ./...
mvn test
pytest

# 构建验证
npm run build
npm run lint
```

**测试执行清单：**

```
□ 单元测试通过
□ 集成测试通过
□ E2E 测试通过（如有）
□ 测试覆盖率达标
□ 手动测试核心流程
□ 边界条件测试
□ 异常情况测试
□ 性能测试（如有要求）
```

## 测试失败处理

```
测试失败时：
1. 记录失败用例和错误信息
2. 分析根本原因
3. 修复代码
4. 重新执行测试
5. 更新测试报告

如果问题无法立即解决：
1. 在测试报告中标注为「阻塞」
2. 说明阻塞原因
3. 等待用户确认是否可以延后处理
```

## 执行流程

```
1. 创建测试计划 (test-plan.md)
2. 编写测试用例 (test-cases.md)
3. 执行测试
4. 生成测试报告 (test-report.md)
5. 处理失败用例
```
